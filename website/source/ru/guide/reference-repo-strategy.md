# Стратегия reference repo

Открывайте эту страницу, когда команда уже понимает, что один repo должен стать эталонным примером пути, но вы хотите, чтобы этот repo учил правильному стандарту, а не случайным привычкам.

## Выбор за 60 секунд

- откройте эту страницу, если один repo вот-вот станет примером, который начнут копировать другие
- откройте её до обновления templates, starter guidance или rollout docs
- откройте её, если repo много, но никто не может договориться, какой из них должен задавать baseline
- не открывайте её, если сам путь ещё не выбран; сначала исправьте это

## Что помогает решить эта страница

- какой repo должен стать reference repo
- что именно этот repo обязан доказать до того, как команда начнёт на него равняться
- как обновлять reference repo и не превращать его в снежинку со своими особыми правилами

## Что такое reference repo

Reference repo — это не просто первый repo, который однажды заработал.

Это repo, который:

- лучше всего представляет выбранную форму продукта
- чисто проходит полный публичный контракт
- достаточно видим для команды, чтобы по нему реально учиться
- достаточно близок к стандартному пути, чтобы его исключения не становились фальшивыми правилами

## Чем reference repo не является

- это не самый сложный repo в парке
- это не repo с наибольшим числом исключений
- это не repo, который один maintainer понимает приватно
- это не место, где пытаются одновременно доказать все target’ы, runtime и delivery model

## Как выбрать хороший reference repo

Выбирайте repo, который:

- достаточно живой, чтобы иметь значение
- достаточно простой, чтобы его можно было объяснить
- действительно представляет путь, который команда хочет копировать
- уже близок к прохождению `doctor -> render -> validate --strict`

Если такого repo нет, не притягивайте выбор за уши. Сначала создайте или очистите один reference repo.

## Что reference repo обязан доказывать

Прежде чем команда начнёт считать repo эталоном, он должен доказывать:

- выбранный target и runtime указаны явно
- rendered outputs воспроизводимы
- CI запускает тот же readiness contract, что и локальная разработка
- README или docs repo объясняют путь без скрытого устного знания
- связанные release notes и support promises всё ещё совпадают с выбранным путём

## Безопасный жизненный цикл

1. Сначала выберите один путь.
   Reference repo не исправит неясный target, runtime или delivery model.
2. Очистите один repo.
   Добейтесь прохождения полного контракта без вручную подправленных generated files.
3. Привяжите публичный baseline.
   Сошлитесь на актуальный release и support rule, на которых стоит этот путь.
4. Дайте другим repo копировать его осознанно.
   Только после очистки repo templates, starter guidance и rollout tracking должны ссылаться на него.
5. Держите его скучным.
   Reference repo должен меняться осторожно, а не становиться самым экспериментальным repo в парке.

## Когда менять reference repo

Меняйте его, когда:

- repo оброс исключениями, привязанными к одному конкретному пути
- команда перешла на другой основной путь
- repo больше не представляет самый чистый публичный baseline

Не меняйте его только потому, что появился более новый repo. Меняйте его тогда, когда текущий reference уже учит не тому стандарту.

## Чего не делать

- не позволяйте самому продвинутому repo случайно стать стандартом
- не обновляйте reference repo так, что это нельзя публично объяснить
- не показывайте команде сразу несколько конфликтующих repo как “эталон”
- не допускайте расхождения между release notes, CI и repo docs в описании выбранного пути

## Финальное правило

Если другой maintainer не может посмотреть на reference repo и понять intended standard без частного объяснения, значит это ещё не настоящий reference repo.
