# Зачем plugin-kit-ai

`plugin-kit-ai` — это система управляемых plugin-проектов для команд, которым нужен один основной репозиторий, выходы под разные target’ы и честная публичная граница поддержки.

Он решает конкретную задачу: помочь команде вести настоящий plugin repo с понятными правилами поддержки, вместо набора вручную правленных target-файлов и одноразовых вспомогательных скриптов.

## Чем это не является

`plugin-kit-ai` — это не:

- магический слой, который делает каждый agent и каждый target одинаково зрелыми
- обещание, что название starter’а задаёт долгосрочную границу репозитория
- замена инженерному решению о том, когда проект уже пора делить по ownership или release cadence

## Что он даёт

- единая управляемая модель проекта вместо постоянного drift в target-файлах
- сильный путь по умолчанию на Go и стабильные локальные пути для Python и Node
- предсказуемые шаги `render` и `validate`
- generated API и support metadata, привязанные к реальным исходным данным

## Кому это подходит

- авторам плагинов, которым нужна более сильная структура, чем ad-hoc local scripts
- командам, которые мигрируют с native target files на управляемую модель проекта
- сопровождающим, которым важны drift detection, strict validation и явные публичные границы

## Когда это плохой выбор

Скорее всего это не ваш инструмент, если:

- вам нужен только маленький одноразовый local script без намерения поддерживать структуру
- вы хотите универсальное управление зависимостями для всех interpreted runtime ecosystems
- вы хотите, чтобы каждый target и каждая hook family имели одинаковые гарантии стабильности

## Главный компромисс

Вы получаете более сильную структуру, более ясные границы и более предсказуемые workflow.

Взамен вы принимаете:

- более жёсткую и осознанную модель проекта
- явные stable/beta boundaries вместо притворства, что всё одинаково зрелое
- рабочий процесс, в котором `render` и `validate --strict` действительно имеют значение
